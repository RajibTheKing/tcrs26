	Figure \ref{fig:overview} illustrates the high-level structure of our system, highlighting the main modules and their interconnections. The \textsf{TrainClient}, runs on the train, collects sensor data from the physical world and reacts upon receiving control commands. The \textsf{RemoteControlClient} is a web application that provides a user interface for operators to render real-time video stream and send control commands. The \textsf{CentralServer} acts as a communication hub between the \textsf{TrainClient} and \textsf{RemoteControlClient}, relaying information between them. In this paper, our focus is on the architecture of the \textsf{TrainClient}, which is implemented using Lingua Franca.

	In Lingua Franca, a reactor is a concurrent, stateful component with well-defined input and output ports, timers, and actions. Reactions are functions triggered by events, execute atomically and deterministically. Lingua Franca enforces a logical time semantics: all events are scheduled and processed in timestamp order, and physical time can be explicitly referenced for real-time interactions. Figure~\ref{fig:train-client} illustrates the top-level components of the \textsf{TrainClient}. The graphical reactor diagrams are generated using the model-based diagramming framework KLighD \cite{SchneiderSvH13} and the Eclipse Layout Kernel \cite{DomroesvHS+23}.

	The \textsf{TrainClient} is the main reactor which instantiates other reactors. The \textsf{VideoStreamer} reactor is responsible for capturing and encoding video frames from physical camera mounted on the train. The encoded video frames are then passed to both \textsf{NetworkProtocol}, and \textsf{DataStorage} reactors. The \textsf{DataStorage} reactor is responsible for storing the video frames and other relevant data locally on the train. The \textsf{NetworkProtocol} reactor handles the connection establishment and data transmission channel with \textsf{Central Server}. Upon receiving data \textsf{NetworkProtocol} forwards them to the Parser reactor which extracts meaningful events. When the parser identifies a new connection establishment request event, it initializes the \textsf{ClockSynchronizer} and the \textsf{HeartbeatHandler}. The \textsf{ClockSynchronizer} is responsible for calculating the clock offset between the train and the remote control client and \textsf{Heartbeathandler} manages the periodic exchange of heartbeat messages to monitor active connectivity. Upon receiving control commands, the \textsf{Parser} forwards them to the \textsf{Controller} reactor which processes the commands and translates them into actionable tasks for physical world and actuate the train accordingly.

	\textsf{HeartbeatHandler} is illustrated in Figure \ref{fig:heartbeat}, is a nice example of how LF's logical time semantics and features like modal reactors can be used to implement a robust connectivity monitoring mechanism. With modal models in LF, reactors enables the coordination of reactive tasks based on modes of operation \cite{SchulzRosengartenvHLB+23c}. In the \textsf{HeartbeatHandler}, we have three modes: \textsf{Init}, \textsf{Send-and-Check}, and \textsf{Stopped}. It works as a finite-state machine within a reactor. Upon initialization, the reactor transitions from \textsf{Init} to \textsf{Send-and-Check} mode, where it periodically sends heartbeat messages using a timer with 500 ms and checks for responses. If a heartbeat message is not received within a specified logical time, currently it is set to 3000 ms, it transitions to the \textsf{Stopped} mode, indicating a loss of connectivity. If a response is received, it remains in the \textsf{Send-and-Check} mode and continues monitoring.
