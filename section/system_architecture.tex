\section{System Architecture}

  	\subsection{Overview}
		This subsection presents an overview of the system architecture, emphasizing its core components and the interactions among them. Figure~\ref{fig:overview} illustrates the high-level structure of the system, highlighting the main modules and their interconnections. The architecture is designed to support multiple trains, each of which connects to a centralized server deployed on a cloud platform through a network connection. Both the trains and other system components can utilize various communication technologies such as 4G, 5G, or Wi-Fi, depending on network availability and operational requirements. Each train, can be equipped with an array of sensors and communication devices that enable continuous data acquisition and reliable transmission to the central server. Similarly, multiple remote control interfaces—accessible through any standard web browser—can connect to the cloud-based server over the flexible networking options. Through these interfaces, an operator can select any train currently connected with the central server, monitor its real-time operational data, and issue control commands. This design ensures robust connectivity, scalability, and real-time management of train operations across distributed environments, independent of specific network infrastructure.

		\subsubsection{Train Client}
			The Train Client module serves as the on-board system can be deployed on any train, responsible for capturing data from camera, preparing telemetry data from sensors, encoding video frames and transmitting this data to the central server while reacts on control commands from remote operators. The Train Client is implemented in Python. The module incorporates several key libraries to achieve its functionality. OpenCV (opencv-python) provides real-time video capture and frame processing capabilities, while PyAV (av) enables hardware-accelerated H.264 video encoding for efficient compression. The aioquic library to integrate the QUIC protocol for low-latency transport, the paho-mqtt client facilitates communication with the MQTT broker, and loguru handles structured logging and diagnostics throughout the application. The encoder component perform real-time H.264 video encoding with dynamic bitrate adaptation capbilities. Train clients only communicate with the central-server. This module integrates three parallel network transport mechanisms. First, a websocket connection is established for any signaling related communication. Second, a QUIC/HTTP3 connection is established for low-latency video streaming.Third, an MQTT connection is established for telemetry data transmission, enabling efficient publish-subscribe messaging between the train and the server. The sensor component is responsible for integrating various sensors on the train to collect data such as speed, location, direction, temperature, and other relevant telemetry information. This component is designed to be modular, allowing for easy addition or removal of sensors as needed. The actuator component handles the execution of control commands received from the remote control consoles via the central server. It interprets commands such as speed adjustments, direction changes, and emergency stops, and translates them into actions performed by the train's control systems. The base class orchestrates the interaction between video capture, encoding, network event handling to ensure smooth operation and data flow within the train client module.
