	Figure \ref{fig:overview} illustrates the high-level structure of our system, highlighting the main modules and their interconnections. The \textsf{TrainClient}, runs on the train, collects sensor data from the physical world and reacts upon receiving control commands. The \textsf{RemoteControlClient} is a web application that provides a user interface for operators to render real-time video stream and send control commands. The \textsf{CentralServer} acts as a communication hub between the \textsf{TrainClient} and the \textsf{RemoteControlClient}, relaying information between them. In this paper, our focus is on the architecture of the \textsf{TrainClient}, which is implemented using Lingua Franca.

	In Lingua Franca, a reactor is a concurrent, stateful component with well-defined input and output ports, timers, and actions. Reactions are functions triggered by events, execute atomically and deterministically. Lingua Franca enforces a logical time semantics: all events are scheduled and processed in timestamp order, and physical time can be explicitly referenced for real-time interactions. Figure~\ref{fig:train-client} illustrates the top-level components of the \textsf{TrainClient}. The graphical reactor diagrams are generated using the model-based diagramming framework KLighD \cite{SchneiderSvH13} and the Eclipse Layout Kernel \cite{DomroesvHS+23}.

	The \textsf{TrainClient} is the main reactor containing all reactors. The \textsf{VideoStreamer} reactor is responsible for capturing and encoding video frames from the physical camera mounted on the train. The encoded video frames are then passed to both \textsf{NetworkProtocol} and \textsf{DataStorage} reactors. The \textsf{DataStorage} reactor is responsible for storing the video frames and other relevant data locally on the train. The \textsf{NetworkProtocol} reactor handles the connection establishment and data transmission channel with the \textsf{CentralServer}. Upon data reception, the \textsf{NetworkProtocol} reactor forwards the received data to the \textsf{Parser} reactor, which parses the payload and classifies it as connection requests, control commands, heartbeat messages, or clock synchronization messages used for clock offset calculation. When the \textsf{Parser} reactor identifies a new connection request, it sends output signals to initialize the \textsf{ClockSynchronizer} and the \textsf{HeartbeatHandler} reactors. The \textsf{ClockSynchronizer} reactor is responsible for calculating the clock offset between the train and the remote control client and \textsf{HeartbeatHandler} reactor manages the periodic exchange of heartbeat messages to monitor active connectivity. Upon receiving control commands, the \textsf{Parser} reactor forwards them to the \textsf{Controller} reactor which processes the commands and translates them into actionable tasks for the physical world and actuate the train accordingly.

	The \textsf{HeartbeatHandler} reactor, shown in Figure \ref{fig:heartbeat}, demonstrates how LF's logical-time semantics and modal reactors can be used to implement a robust connectivity monitoring mechanism. With modal models, reactors enable the coordination of reactive tasks based on modes of operation \cite{SchulzRosengartenvHLB+23c}. The \textsf{HeartbeatHandler} operates in three modes: \textsf{Init}, \textsf{Send-and-Check}, and \textsf{Stopped}. During initialization, the reactor transitions from \textsf{Init} to \textsf{Send-and-Check} and schedules a logical action with delay of 3000~ms, within which a new heartbeat message is expected from the remote control client. While in the \textsf{Send-and-Check} mode, one reaction uses a timer with a logical time of 500~ms to transmit heartbeat messages periodically. A separate reaction monitors the reception of heartbeat messages from the remote control client. Upon receipt of a new heartbeat message, it updates the heartbeat counter and reschedule the logical action. If a heartbeat stop signal is received, or the delay window expires without receiving a new heartbeat message, the reactor transitions to the \textsf{Stopped} mode. This transition indicates either a loss of connectivity or the absence of an active remote control client.
