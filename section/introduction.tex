\section{Introduction}
\label{sec:intro}
    The rapid technological advancement in railway systems is driving a fundamental shift from manual human control to autonomous operations. This transformation is particularly evident in the pursuit of higher grades of automation, i.e. GoA levels 3 and 4, where trains operate with minimal human intervention or become completely driverless. As railway systems progress toward full automation, the role of remote control systems becomes increasingly critical. The Remote control system is a vital layer in railway operations, providing essential fallback capabilities when autonomous systems encounter failures \cite{Shift2Rail2019, CoganMilius2023}.

    To fulfill this critical fallback role effectively, a remote control system must support two fundamental communication channels: one for transmitting control commands and another for streaming real-time video. In most existing teleoperated systems, control commands are typically transmitted using reliable protocols such as MQTT or WebSocket over TCP, while video streaming is handled by UDP-based protocols like RTSP or WebRTC \cite{10597107, 11416303, 11494127}.
    TCP's strict in-order deliver causes Head-of-Line (HoL) blocking. A single lost packet stalls subsequent control commands until retransmission occurs, introducing unpredictable latency spikes \cite{Langley2017QUIC}. WebRTC, while designed for low-latency media streaming, add extra layer of architectural complexity by requiring a separate signaling server and startup latency for connection establishment \cite{Mejias2025QUICWebRTC}.

    WebTransport is built on the QUIC protocol, standardized as the foundation for HTTP/3, a promising alternative to both WebSocket and WebRTC protocols for time-centric reactive software systems. It offers supports for both reliable streams for lossless control commands and unreliable datagrams for latency-sensitive video streaming, making it well-suited for remote control applications in railway operations.\cite{10.1145/3696355.3699698, rfc9000}.

    Remote Control System includes physical components such as sensors and actuators. Data exchange and coordination among these components are commonly implemented using mechanisms such as the publish–subscribe pattern or callback-based execution. While widely adopted, these approaches can introduce non-deterministic behavior due to unpredictable callback ordering and concurrent event handling. To achieve deterministic execution in this cyber-physical environment, the train client is implemented using Lingua Franca, a reactor-oriented coordination language that guarantees determinism by construction. Lingua Franca decouples logical time from physical wall-clock time, enabling predictable and reproducible execution semantics \cite{10.1145/3448128}. This design eliminates race conditions and the non-deterministic callback ordering often associated with traditional publish–subscribe mechanism, thereby ensuring consistent and predictable execution traces on the physical train system \cite{teper2026deterministicexecutionros2}.

    
    This paper provides following contribution:
        \begin{itemize}
            \item The architectural design of a train client using Lingua Franca
            
            \item A performance analysis of WebTransport in the context of remote control system under realistic network conditions.
        \end{itemize}
