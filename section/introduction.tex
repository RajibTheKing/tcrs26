\section{Introduction}
\label{sec:intro}
    The rapid technological advancement in railway systems is driving a fundamental shift from manual human control to autonomous operations. This transformation is particularly evident in the pursuit of higher grades of automation, i.e. GoA levels 3 and 4, where trains operate with minimal human intervention or become completely driverless. As railway systems progress toward full automation, the role of remote control systems becomes increasingly critical. The remote control system is a vital layer in railway operations, providing essential fallback capabilities when autonomous systems encounter failures \cite{Shift2Rail2019, CoganMilius23}.

    To fulfill this critical fallback role effectively, a remote control system must support two fundamental communication channels: one for transmitting control commands and another for streaming real-time video. In most existing teleoperated systems, control commands are typically transmitted using protocols such as MQTT\footnote{https://mqtt.org/} or WebSocket\footnote{https://datatracker.ietf.org/doc/rfc6455/}, while video streaming is handled by UDP-based protocols such as RTSP\footnote{https://datatracker.ietf.org/doc/rfc7826/} or WebRTC\footnote{https://webrtc.org/} \cite{MejiasFVALD24, SunTsai25, FezeuCZDVBS+26}. TCP's strict in-order deliver causes Head-of-Line (HoL) blocking. A single lost packet stalls subsequent control commands until retransmission occurs, introducing unpredictable latency spikes \cite{LangleyRWVKZ+17}. WebRTC, while designed for low-latency media streaming, add extra layer of architectural complexity by requiring a separate signaling server and startup latency for connection establishment \cite{MejiasYMVA+25}.

    WebTransport, which is built on the QUIC protocol, standardized as the foundation for HTTP/3, is a promising alternative to both WebSocket and WebRTC protocols for time-centric reactive software systems. It offers supports for both reliable streams for lossless control commands and unreliable datagrams for latency-sensitive video streaming, making it well-suited for remote control applications in railway operations \cite{AmetTS25, rfc9000}.

    A remote control system includes physical components such as sensors and actuators. Data exchange and coordination among these components are commonly implemented using mechanisms such as the publish–subscribe pattern or callback-based execution. While widely adopted, these approaches can introduce non-deterministic behavior due to unpredictable callback ordering and concurrent event handling. To achieve deterministic execution in such cyber-physical system, we propose to implement the software component using Lingua Franca that runs on the train. This component handles network communication, control logic, and interaction with onboard sensors and actuators. Lingua Franca is a reactor-oriented coordination language that guarantees determinism by construction. It decouples logical time from physical wall-clock time, enabling predictable and reproducible execution semantics \cite{LohstrohMBL21}. This design eliminates race conditions and the non-deterministic callback ordering often associated with traditional publish–subscribe mechanism.


    This paper provides following contributions:
        \begin{itemize}
            \item The architectural design of the train controller and remote control receiver using Lingua Franca.

            \item We evaluate the feasibility of using QUIC based WebTransport in the context of the remote control system which is tested on real world network conditions.
        \end{itemize}
